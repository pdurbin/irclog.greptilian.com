16:51 sjmaple what happens if the coordinator or remote coordinator goes down? Also, who determines which node is the coordinator?
16:52 snazy there's no "the coordinator". coordinator is just the node that executes the request - any node can be the coordinator for any query
16:53 snazy in terms of performance (less network roundtrips) the drivers prefer a replica node to execute a query.
16:53 jkremser so consistency lvl denotes how many acks does it needs to respond to the client?
16:53 sjmaple ahh ok, how does a coordinator pick which remote coordinator to use?
16:53 snazy sjmaple: same algorithm :)
16:53 snazy just in other DC
16:53 snazy jkremser: yes
16:53 sjmaple ok, how does it know of them?
16:54 sjmaple some multicast like thing?
16:54 jeremy_prime joined #virtualJUG
16:54 snazy no mc - just tcp connections. there's a (small) set of nodes in the config (= seed nodes) - these are "initial contact points"
16:55 sjmaple ahh ok, got it
16:55 snazy from these the nodes know about the other nodes (and their token-ranges)
16:55 snazy but: seed-nodes != master :)
16:55 sjmaple :)
16:55 sjmaple sounds like master with failover to me ;)
16:55 sjmaple mwahaha
16:57 snazy they're just needed on startup - after that, nodes "gossip" about the other nodes
16:57 sjmaple like sharing a live nodes table or similar I guess
16:59 snazy gossip data (on the wire) is really small. but in the end it builds a complete view of the whole cluster in every node
