22:00 sfisque alexczar the real question is "how often will they change".  if you put them in a properties file, they're still wired to a build unless you're deploying as an exploded ear/war (which has it's own security issues).  if they are fairly glacial then yah, prop file will do.  if they're going to change fairly briskly, id' put them in the database.  a simple key, url table would do fine
22:02 sfisque you could even bind them into jndi and expose them as @Resources.   additionally, you could make them exposed via a JMX bean, that way they can be updated real-time via a jmx endpoint and serialize the data into either database or a SFSB that passivates and never goes out of scope (inject a reference into an application scoped bean)
